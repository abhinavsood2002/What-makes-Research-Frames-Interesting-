# Frame Ranking System - Comprehensive Documentation

## Overview
The Frame Ranking System uses a modified Swiss Tournament algorithm to efficiently rank research frames through head-to-head comparisons. This system dramatically reduces the number of comparisons needed while maintaining statistical confidence in the final rankings.

## Algorithm: Modified Swiss Tournament System

### Core Principles
1. **Pairing Strategy**: Uses round-robin rotation to ensure diverse matchups
2. **Comparison Reduction**: Reduces O(n²) all-pairs comparisons to approximately O(n log n)
3. **Fairness**: Each frame gets roughly equal opportunities to compete
4. **Bias Prevention**: No visual bias in comparisons (neutral styling)

### Frame Count Specifications
- **Minimum Frames**: 8 (statistical requirement for meaningful rankings)
- **Maximum Frames**: 15 (complexity management and user experience)
- **Optimal Range**: 10-12 frames (best balance of thoroughness and efficiency)

### Comparison Count Calculations

For N frames, the system generates approximately:
- **Minimum Comparisons**: N × 1.5
- **Maximum Comparisons**: N × 2.5
- **Target Range**: N × 2.0

#### Specific Examples:
- **8 frames**: ~16 comparisons (vs 28 for all-pairs)
- **10 frames**: ~20 comparisons (vs 45 for all-pairs)
- **12 frames**: ~24 comparisons (vs 66 for all-pairs)
- **15 frames**: ~30 comparisons (vs 105 for all-pairs)

## Pairing Generation Algorithm

### Round-Robin with Rotation Method

1. **Initial Setup**: All frames are arranged in a list
2. **Round Generation**: For each round, frames are split into two halves
3. **Rotation Logic**: Right half rotates by round number to create new pairings
4. **Conflict Avoidance**: System tracks previous pairings to avoid repeats
5. **Odd Frame Handling**: With odd frame counts, one frame sits out each round (rotated fairly)

### Mathematical Foundation

For N frames across R rounds:
- **Rounds Required**: ceil(log₂(N)) minimum for statistical confidence
- **Target Rounds**: min(4, N-1) for optimal balance
- **Pairings per Round**: floor(N/2)
- **Total Pairings**: R × floor(N/2)

### Example Pairing Sequence (10 frames)

**Round 1**: [1v6, 2v7, 3v8, 4v9, 5v10]
**Round 2**: [1v7, 2v8, 3v9, 4v10, 5v6] (right half rotated)
**Round 3**: [1v8, 2v9, 3v10, 4v6, 5v7] (right half rotated again)
**Round 4**: [1v9, 2v10, 3v6, 4v7, 5v8] (final rotation)

## Ranking Calculation

### Win-Based Scoring System
1. **Simple Wins**: Each victory counts as +1 point
2. **Tie Breaking**: Frame ID used for consistent tie resolution (lower ID wins ties)
3. **Final Ranking**: Sorted by wins (descending), then by ID (ascending)

### Statistical Confidence
- **Minimum Comparisons**: Each frame competes in at least 40% of possible matchups
- **Balanced Exposure**: No frame gets significantly more or fewer comparisons
- **Transitivity Validation**: System maintains logical consistency in rankings

## Bias Prevention Measures

### Visual Neutrality
- **No Color Coding**: Both comparison options use identical gray styling
- **Position Randomization**: Left/right positioning doesn't indicate preference
- **Identical Interactions**: Both cards have the same hover and click behaviors
- **Content Focus**: Only the actual frame content differs between options

### Algorithmic Fairness
- **Random Sampling**: If >15 frames are marked 'interesting', random selection ensures fairness
- **Rotation Equality**: Each frame sits out equal rounds (for odd counts)
- **Pairing Distribution**: Algorithm ensures roughly equal exposure for all frames

## Performance Characteristics

### Time Complexity
- **Pairing Generation**: O(R × N) where R is rounds, N is frames
- **Comparison Processing**: O(C) where C is total comparisons
- **Final Ranking**: O(N log N) for sorting

### Space Complexity
- **Pairing Storage**: O(R × N/2) for all pairings
- **Conflict Tracking**: O(N²) for played-against matrix
- **Result Storage**: O(N) for final rankings

### User Experience
- **Session Duration**: ~5-10 minutes for 10 frames
- **Cognitive Load**: Manageable number of decisions
- **Progress Tracking**: Clear indication of completion percentage

## Limitations and Assumptions

### Statistical Limitations
1. **Sample Size**: Requires minimum 8 frames for statistical validity
2. **Transitivity**: Assumes preferences follow transitive properties
3. **Consistency**: User judgments assumed to be consistent across session

### Implementation Constraints
1. **Session Persistence**: Rankings are session-based, not longitudinal
2. **User Fatigue**: Optimal range prevents decision fatigue
3. **Computational Limits**: Maximum 15 frames prevents exponential complexity

### Known Edge Cases
1. **Perfect Ties**: Resolved by frame ID (deterministic but arbitrary)
2. **Inconsistent Preferences**: No cycle detection implemented
3. **Incomplete Sessions**: No partial ranking recovery mechanism

## Algorithm Validation

### Theoretical Foundation
- **Based on**: Swiss tournament system used in chess and competitive gaming
- **Adapted for**: Research frame evaluation with bias prevention
- **Validated through**: Comparison count reduction and ranking accuracy

### Comparison with Alternatives

**All-Pairs Comparison**
- Pros: Complete information, perfect accuracy
- Cons: O(n²) complexity, user fatigue, impractical for >6 frames

**Single Elimination**
- Pros: Minimal comparisons
- Cons: High variance, single-point failures, unfair to strong frames

**Random Sampling**
- Pros: Extremely fast
- Cons: Poor statistical confidence, high variance

**Modified Swiss (Current)**
- Pros: Balanced complexity/accuracy, proven tournament system, bias-free
- Cons: Still requires multiple rounds, assumes transitivity
